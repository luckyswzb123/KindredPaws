# 萌爪家园 (Kindred Paws) - PC端后台管理系统开发规划

## 1. 项目定位与原则
* **独立性**：本后台管理系统（Admin Dashboard）将作为一个完全独立的项目/模块进行开发。不会侵入、不修改现有移动端 App 的代码（建议开辟新的 `admin` 前端模块和独立的后端 API 路由层）。
* **风格一致性**：管理后台的 UI 风格将完全延续 App 的视觉规范，包括主品牌色（Accent 色系）、圆角设计（12px/16px）、Glassmorphism（毛玻璃效果阴影）和流畅的动效，但针对 PC 宽屏进行布局优化（例如使用左侧菜单侧边栏 + 右侧数据内容区）。

## 2. 技术栈选型
* **前端框架**：React + Vite + TypeScript (与 C 端技术栈相同)
* **路由管理**：React Router DOM (适配 PC 端宽屏的嵌套侧边栏路由)
* **样式方案**：Tailwind CSS (直接复用现有 App 的 `tailwind.config.js` 的配色体系，如 `accent`, `surface-container` 等)
* **组件图标**：Lucide React
* **文件存储**：**Cloudflare R2** (兼容 S3 协议，提供全球 CDN 加速对象存储，专门用于解决宠物多图和高容量诉求)
* **后端 API**：Node.js + Express (独立于现有面向 C 端的路由，新增 `/api/admin/*` 专属路由组，受后台鉴权中间件保护)
* **数据库**：Supabase (PostgreSQL)
* **鉴权服务**：Supabase Auth (基于 JWT 的身份验证及跨端 Session 管理)

## 3. 核心功能及实现逻辑

### 3.1 录入宠物信息并接入 Cloudflare R2
* **需求表现**：在后台填写包含宠物名字、品种、年龄、健康状态等表单并上传图片。
* **业务执行流**：
  1. 后台前端选择图片后，调用专门的图片直传接口或通过 `FormData` 上传给独立后端接口 `/api/admin/upload`。
  2. 后端利用 `@aws-sdk/client-s3` 承接此图片流，将文件上传至 Cloudflare R2 Bucket。
  3. R2 保存成功后，后端组装并返回带公网访问加速域名的 URL。
  4. 后台收到完整的 URL 和表单后，将其插入 Supabase 的 `pets` 表中。

### 3.2 查看所有宠物信息
* **需求表现**：以数据表格 (Data Table) 或信息网栏形式详细展示所有宠物数据。
* **业务执行流**：
  * 请求 `/api/admin/pets` 获取所有（含下线的）宠物信息，支持搜索、筛选。后台将清楚展示“在线”或“已下线/已领养”标志。

### 3.3 领养申请审批及联动逻辑
* **需求表现**：后台管理员可以看到所有用户的领养申请，并能操作“审批通过”或“一键拒绝”。
* **业务执行流（审批通过）**：
  1. 更改 `applications` 表的状态为 `approved`。
  2. 向系统生成 `messages` 表数据。App 用户将在【收件箱】获得带 `content` 的提示，告知审批通过。
  3. **宠物下线（App 端不可见，Admin 可见）**：更新 `pets` 表的状态（例如新增一个 `is_published: false` 的标识，或者将 `status` 标记为 `adopted`），这样 App 端的 API 接口拉取时会自动过滤此宠物（从 App 下线）。但后台拉取所有的包含被领养宠物的全部数据不受影响。

### 3.4 管理员登录与鉴权流程
* **身份认证**：后台管理系统接入 Supabase Auth 提供基于 E-mail/Password 的标准身份验证。
* **用户信息存储**：所有用户信息（含管理员）存储在 Supabase。管理员权限通过在 `user_profiles` 表中增加 `is_admin: boolean` 字段或在 `auth.users` 的 `raw_user_meta_data` 中通过 Role 标记。
* **业务执行流**：
  1. **登录**：用户在后台专用登录页输入凭据。前端调用 `/api/auth/login` 或直接通过 Supabase Client 完成认证。
  2. **令牌管理**：认证通过后，Supabase 返回 `access_token`。前端将其存储在 `localStorage` 或 `sessionStorage` 中，并在后续请求头中带上 `Authorization: Bearer <token>`。
  3. **后端校验**：`/api/admin/*` 专属路由组受 `requireAdminAuth` 中间件保护。
     - **第一层校验**：解析 JWT，验证令牌有效性及用户 ID。
     - **第二层校验**：查询数据库，确认该用户是否具备管理员（Admin）权限，非管理员用户即使登录也无法调用 API。
  4. **前端拦截**：利用 React Router 的 `Navigate` 组件进行全局路由保护。未登录或权限不足时直接跳转至 `/login` 页。

## 4. 架构实施与目录规划

为了满足**“相互独立且不修改现有代码”**的要求，我们将在项目根目录创建隔离结构：

```text
e:\Project\Kindred.Paws\
├── src/                (C端 App 前端，保持完全隔离，不做改动)
├── server/             (后端服务栈)
│   ├── routes/
│   │   ├── admin.ts    (新增: 独立为管理系统定制的后台 API)
│   │   ├── upload.ts   (新增: 连接 R2 处理图片上传逻辑)
├── admin/              (全新创建: 独立的 PC 后台前端工程)
│   ├── index.html
│   ├── vite.config.ts
│   ├── package.json
│   ├── src/
│   │   ├── pages/      (Login, Dashboard, Pets, Applications 等)
│   │   ├── components/ (Sidebar, Navbar, ProtectedRoute, Table 等复刻风格组件)
│   │   ├── lib/        (supabase.ts, api.ts 等核心类库)
```

## 5. 开发里程碑划分

* **第一阶段 (Phase 1)：独立前端骨架搭建**
  * 在 `/admin` 中初始化 Vite React TypeScript 工程。
  * 复刻并植入与 App 一致的 Tailwind 变量和设计语言。
  * **权限模块实现**：构建 `Login` 页面及 Supabase Auth 登录逻辑。
  * 构建 PC 端独有的侧边导航栏及页面路由架构（含 `ProtectedRoute` 拦截器）。
* **第二阶段 (Phase 2)：Cloudflare R2 集成与录入**
  * 基于您提供的 Cloudflare R2 Credentials（S3 URL、Access ID、Secret），打通 `upload.ts`，完成存储环境调试。
  * 编写 `admin` 中的新增宠物可视化表单面板。
* **第三阶段 (Phase 3)：独立后端 API 与逻辑完善**
  * 实现 `/api/admin/*` 的数据库权限分离读取能力。
  * 编写所有审批、状态联动、系统发信业务，并完成前后端最终对接。
